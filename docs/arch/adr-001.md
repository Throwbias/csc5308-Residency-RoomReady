# ADR-001: Frontend-Only Architecture with Local Storage for POC

**Status:** Proposed
**Date:** June 13, 2026

## Context
The engineering team is tasked with building a Proof of Concept (POC) for the RoomReady study room reservation system within a strict 5-hour Saturday sprint. The primary goal of the POC is to demonstrate the core user reservation flow and immediate UI status updates during Sunday's live demo. We need an architecture that minimizes configuration overhead while reliably showcasing the system's intended behavior.

## Decision
We decided to build a frontend-only web application using HTML5, CSS3, Vanilla JavaScript (ES6), and browser `localStorage` for state management. We will use simulated frontend data (hardcoded JSON/JS objects) rather than a production database to track room availability and bookings. 

## Alternatives Considered
* **Full-Stack Implementation (e.g., React + Node.js + PostgreSQL):** Evaluated for production realism but rejected. Setting up the database schema, API routing, environment variables, and backend hosting would consume the majority of the 5-hour sprint, leaving insufficient time to polish the user interface and demo flow.
* **Frontend Framework (e.g., React, Vue, or Angular):** Evaluated for component reusability. Rejected because the POC scope is small enough that Vanilla JS is sufficient. Avoiding Node package managers (npm), build steps, and compilers eliminates potential troubleshooting bottlenecks during the sprint.

## Consequences
* **What this makes easier:** Immediate local setup (zero dependencies), guaranteed uptime for the live demo (runs directly in the browser), and rapid iteration of the UI and core logic.
* **Trade-offs introduced:** The system cannot handle multiple concurrent users across different devices. State persistence is tied strictly to the local browser session (`localStorage`), meaning data will be wiped if the cache is cleared. The system will require a complete backend integration rewrite for the eventual production release.
